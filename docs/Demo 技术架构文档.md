# 招聘流程自动化 Demo 技术架构文档

## 1. 文档目标

本文档用于定义招聘流程自动化 Demo 的整体技术架构，明确本期 Demo 的技术选型、系统分层、模块职责、核心数据流和本地运行方式。

本期目标不是构建完整 ATS，也不是解决职位自动发布和候选人自动抓取问题，而是在本地环境下完成一个可运行、可演示、可扩展的全栈原型，重点验证以下能力：

- 工作描述的 AI 辅助生成与修改
    
- 候选人资料的导入、整理与入库
    
- 候选人 AI 分析、打分、排序与展示
    
- 人工反馈管理与邮件草稿生成
    

---

## 2. 总体架构目标

本项目的技术架构需要满足以下几个目标：

### 2.1 本地可独立运行

所有服务默认运行在开发者本地主机，不依赖云数据库、对象存储或外部任务系统。

### 2.2 架构尽量简单

Demo 阶段优先保证链路打通、界面可演示、数据可落库，不引入不必要的分布式组件。

### 2.3 为后续扩展预留空间

虽然本期只在本地运行，但系统设计应尽量避免把未来扩展路径堵死，例如：

- SQLite 平滑切换到 PostgreSQL
    
- 本地文件存储切换到对象存储
    
- 同步 AI 处理切换到异步任务队列
    
- Web 端继续扩展到移动端
    

### 2.4 前端从第一版开始支持响应式布局

本期虽然只做 Web Demo，但页面结构不能写死为桌面端固定布局，需要为未来移动端适配保留空间。

---

## 3. 技术选型

## 3.1 前端

前端采用：

- **Next.js**
    
- **TypeScript**
    
- **Tailwind CSS**
    

选择理由如下：

Next.js 适合快速搭建以表单、列表、详情页和管理后台为主的 Web 应用，能较快完成 Demo 所需界面。  
TypeScript 有利于前后端接口字段的清晰约束。  
Tailwind CSS 适合从第一版开始就做响应式布局，便于后续继续适配不同窗口尺寸和移动端。

前端职责主要包括：

- 职位信息录入与 JD / rubric 编辑页面
    
- 候选人导入页面
    
- 候选人列表与详情展示
    
- AI 分析结果展示
    
- 反馈录入与邮件草稿展示
    

---

## 3.2 后端

后端采用：

- **FastAPI**
    
- **Python 3.11+**
    

选择理由如下：

本项目中 AI 分析流是核心能力之一，而 LangChain 和 OpenAI 接入在 Python 生态中更自然。  
FastAPI 适合快速实现本地 API 服务，接口定义清晰，调试方便，便于后续把业务 API 和 AI 工作流统一承载在一个服务内。

后端职责主要包括：

- 提供业务 API
    
- 处理 Job 定义生成与 rubric 评估规范维护
    
- 处理候选人导入
    
- 执行简历解析与数据标准化
    
- 调用 LangChain + OpenAI 完成 rubric 逐项分析与 supervisor 汇总
    
- 生成硬门槛判断、子分、标签、推荐结论和证据点
    
- 保存反馈与邮件草稿
    

---

## 3.3 数据库

数据库采用：

- **SQLite**
    

选择理由如下：

SQLite 适合本地 Demo，部署成本低，无需单独启动数据库服务，便于直接以单文件形式管理数据。  
当前系统的数据规模较小，业务以中低并发、单机本地演示为主，SQLite 足够支撑。

本期数据库主要用于存储：

- 职位信息
    
- 候选人基础信息
    
- 候选人与职位的关联
    
- AI 分析结果
    
- 人工反馈
    
- 邮件草稿
    
- 成功入库候选人的文件路径
    

---

## 3.4 ORM 与数据访问

建议采用：

- **SQLAlchemy 2.0**
    
- **Alembic**
    

原因是：

SQLAlchemy 更适合作为后续可扩展的 ORM 基础，后面如果数据库从 SQLite 切换到 PostgreSQL，迁移成本相对更低。  
Alembic 用于维护数据库 schema 变更，避免即使是 Demo 项目也完全依赖手工建表。

如果你更想追求 Demo 开发速度，也可以接受 SQLModel，但从长期可维护性和灵活性来看，我更建议直接用 SQLAlchemy。

---

## 3.5 AI 能力层

AI 模块采用：

- **LangChain**
    
- **OpenAI API**
    

这一层主要负责：

- JD 文本生成与改写
    
- 候选人简历信息归纳
    
- 候选人与职位匹配分析
    
- 标签提取
    
- 推荐理由生成
    
- 邮件回复草稿生成
    

本期不做复杂 agent，也不做自治式 AI 调度。  
AI 层应优先做成**可控、可解释、结构化输出**的 pipeline。

---

## 3.6 文件存储

本期文件存储采用：

- **本地目录存储**
    

系统会将上传的导入文件、简历 PDF 等先放入本地临时目录处理；只有分析成功后，数据库中才保存：

- 文件路径
- 最终文件名
    
- 文件类型
    
- 关联对象
    

这样做的好处是：

- 实现简单
    
- 本地可直接查看原始文件
    
- 后续可替换为对象存储而不需要完全重写业务逻辑
    
- Demo 阶段失败时可直接清理临时文件，不引入中间状态持久化

## 4. 系统总体架构

系统整体采用典型的前后端分离单体架构。

### 4.1 架构组成

系统由以下几个部分构成：

#### 前端 Web 应用

负责界面展示、用户交互、表单提交、列表筛选和详情查看。

#### FastAPI 后端服务

负责业务逻辑处理、文件上传、数据持久化、AI 调用和统一 API 输出。

#### SQLite 数据库

负责结构化数据的本地持久化。

#### 本地文件目录

负责保存成功入库候选人的原始文件，并在处理期间承载临时文件。

#### OpenAI / LangChain AI 层

负责文本理解、生成和结构化分析。

---

## 5. 模块划分

## 5.1 职位管理模块

该模块负责职位信息维护、JD 生成与编辑，以及 rubric 评估规范维护。

主要职责：

- 创建职位
    
- 编辑职位基础信息
    
- 调用 AI 生成 JD 草稿
    
- 生成初始 rubric 草稿
    
- 对 JD 与 rubric 进行 AI 改写
    
- 保存职位版本
    

这一模块是候选人分析的上游输入模块。

---

## 5.2 候选人导入模块

该模块负责接收人工导出的候选人资料，并将其整理后入库。

主要职责：

- 上传候选人导入文件
- 拦截浏览器默认拖拽打开文件行为
    
- 读取文件内容
    
- 提取候选人信息
    
- 执行基础标准化
    
- 保存候选人记录
    
- 建立候选人与职位的关联
    

本期默认是人工导出后再上传，不负责平台自动采集。

---

## 5.3 候选人分析模块

该模块负责基于职位信息和候选人资料执行 AI 分析。

主要职责：

- 读取职位信息
    
- 读取候选人信息和简历文本
    
- 构造标准化输入和 rubric 逐项分析输入
    
- 调用 LangChain + OpenAI
    
- 生成 Candidate 标准化结果
    
- 生成 rubric 逐项分析结果
    
- 生成 supervisor 汇总结果
    
- 保存标签、总分、推荐结论和证据点
    
- 提供展示所需结果
    

这是本项目的核心价值模块。

---

## 5.4 反馈管理模块

该模块负责承接 HR 或业务方的人工判断。

主要职责：

- 记录候选人状态
    
- 记录人工备注
    
- 保存人工标签
    
- 形成候选人处理历史
    

这个模块的意义在于让系统不只是展示 AI 结论，还能承接实际业务操作。

---

## 5.5 邮件草稿模块

该模块负责根据反馈结果生成后续邮件草稿。

主要职责：

- 根据候选人状态生成邮件类型
    
- 基于候选人和职位信息生成草稿
    
- 展示主题与正文
    
- 保存邮件草稿记录
    

本期只生成草稿，不做真实发送。

## 6. 逻辑分层设计

后端建议按以下方式分层。

## 6.1 API 层

负责定义 HTTP 接口，接收请求并返回响应，不直接承载复杂业务逻辑。

例如：

- 职位相关接口
    
- 候选人导入接口
    
- 分析触发接口
    
- 反馈保存接口
    
- 邮件草稿生成接口
    

---

## 6.2 Service 层

负责组织业务逻辑，是系统的核心应用层。

例如：

- JobService
    
- CandidateImportService
    
- CandidateAnalysisService
    
- CandidateScoringService
    
- CandidateSupervisorService
    
- FeedbackService
    
- EmailDraftService
    

Service 层负责协调数据库访问、文件处理和 AI 调用。

---

## 6.3 Repository 层

负责与数据库交互，封装基础数据读写逻辑。

这一层的职责是：

- 屏蔽具体 ORM 调用细节
    
- 使业务层不直接依赖底层 SQL 细节
    
- 为未来替换数据库或做更复杂查询保留空间
    

---

## 6.4 AI Workflow 层

单独抽离 AI 调用逻辑，不要把 prompt 和模型调用直接散落在 Service 中。

这一层主要负责：

- prompt 模板管理
    
- Candidate 标准化输入组装
    
- rubric 逐项分析输入组装
    
- LangChain 调用
    
- 子 agent 调用编排
    
- supervisor 汇总调用
    
- 输出结构校验
    
- 结果后处理
    

建议这一层尽量保证输入输出稳定，避免“同一个接口每次返回字段都不一致”。

---

## 6.5 文件处理层

该层负责导入文件、简历文件等的存储与读取。

主要职责：

- 文件命名与落盘
    
- 文件路径管理
    
- 文本抽取
    
- 导入文件解析
    

---

## 7. 前端架构设计

前端以页面展示和交互为主，不承担复杂业务逻辑。

## 7.1 页面层

建议最少包含以下页面：

- 职位列表页
    
- 职位详情 / 编辑页
    
- 候选人导入页
    
- 候选人列表页
    
- 候选人详情页
 该页包含原始输入、标准化信息、rubric 逐项分析、supervisor 汇总，以及反馈与邮件草稿操作区
    

---

## 7.2 组件层

前端组件应尽量分为：

- 页面级容器组件
    
- 通用业务组件
    
- 基础 UI 组件
    

避免把复杂逻辑直接写死在单个页面文件中。

---

## 7.3 状态管理

本期 Demo 不需要非常重的全局状态管理。

建议优先采用：

- 页面局部状态
    
- React Query 管理服务端数据获取与缓存
    

这样结构更轻，也更适合管理后台型应用。

---

## 7.4 响应式布局原则

虽然本期只做 Web Demo，但从第一版开始应遵守以下原则：

- 布局不依赖固定屏宽
    
- 列表和详情区域支持垂直堆叠
    
- 侧边导航在小屏下可折叠
    
- 表单区块支持多行布局
    
- 候选人卡片和筛选区可在窄屏下重排
    

本期不要求专门设计移动端交互，但要避免未来移动适配时出现大规模返工。

## 8. 数据流设计

## 8.1 职位创建与 JD 生成流程

用户在前端输入职位基础信息。  
前端调用后端职位接口。  
后端保存基础职位数据，必要时调用 AI 生成 JD 草稿和初始 rubric。  
首次生成成功后，系统先创建一条 `draft` 状态的 Job 记录。  
生成后的内容返回前端展示，并在 B 页面中继续编辑 JD 与 rubric。  
用户点击“完成”后，后端基于当前编辑区内容再次调用一次最终定稿 LLM。  
只有最终定稿成功并通过结构校验后，才将 Job 当前生效内容正式落库。

---

## 8.2 候选人导入流程

用户上传候选人导出文件或简历文件。  
后端接收文件并放入临时处理目录。  
后端解析文件内容，提取候选人基础信息。  
系统执行基本标准化和去重判断。  
只有全部处理成功后，才将候选人数据与职位关联并写入数据库。  
任一步失败则直接返回错误，并清理临时文件，不保留 Candidate 相关中间记录。

---

## 8.3 候选人 AI 分析流程

用户在前端触发分析，或系统在导入后直接执行分析。  
后端读取职位数据和候选人数据。  
AI Workflow 层先完成 Candidate 标准化。  
随后按 Job rubric 拆分分析任务，对每个维度执行独立子 agent 分析。  
所有子 agent 完成后，再由 supervisor 汇总硬门槛结果、普通项结果、总分、标签、推荐结论和证据点。  
系统对结果进行校验和后处理。  
最终将标准化结果、逐项分析结果和 supervisor 汇总结果保存入库，并返回前端展示。

---

## 8.4 反馈与邮件草稿流程

HR 在前端查看候选人详情。  
录入反馈或修改状态。  
后端保存反馈记录。  
如需要生成邮件草稿，则调用 AI Workflow 生成文本内容并保存。  
前端展示生成结果供用户查看。

---

## 9. 数据模型设计原则

本期不在本文档中细化具体表字段，但建议至少围绕以下核心对象建模：

- Job
    
- Candidate
    
- CandidateProfile
    
- CandidateRubricResult
    
- CandidateTag
    
- Feedback
    
- EmailDraft
    
- FileAsset
    

建模原则如下：

### 9.1 原始数据与分析结果分离

候选人原始信息和 AI 生成结果不要混在一张表里，避免后续分析结果重跑时污染原始数据。

### 9.2 候选人与职位关系单独建模

同一个候选人未来可能对应多个职位，因此建议将候选人与职位关系单独建模。

### 9.3 文件与业务对象解耦

文件系统负责保存文件实体，数据库只负责记录其路径和归属关系。

### 9.4 保留人工反馈独立记录

人工反馈不要直接覆盖 AI 输出，要单独留存。

### 9.5 Job draft 与正式生效内容分离

新建 Job 的首次生成可以先创建 `draft` 记录，供 B 页面继续编辑。  
`draft` 不视为正式入库的生效 Job。  
若用户在新建流程点击取消，则删除对应 `draft`；系统也可定期清理长时间未完成的 `draft` 记录。

## 10. AI 架构设计原则

## 10.1 本期采用固定多阶段 Workflow，而不是自治式 Agent

本期 AI 处理流建议使用固定流程：

- 输入准备
    
- Candidate 标准化
    
- rubric 逐项子 agent 分析
    
- supervisor 汇总
    
- 模型调用
    
- 结构化输出
    
- 落库展示
    

本期允许使用按 rubric 拆分的子 agent 和 supervisor 汇总结构，  
但不引入开放式自治 agent、工具调用自治或不受控的多轮决策逻辑。

---

## 10.2 输出必须尽量结构化

AI 输出应尽量是可解析、可落库、可展示的结果，例如：

- 标签列表
    
- 推荐结论
    
- 推荐理由
    
- 若干可比较的 rubric 子项结果
    
- 硬门槛判断
    
- 核心证据点
    

这样比只生成一段自然语言更适合 Demo 展示和后续扩展。

---

## 10.3 AI 结果需要可解释

Demo 的重点不是追求“绝对准确”，而是让 AI 结果足够有展示价值并便于人工理解。  
因此逐项分析理由、硬门槛结果、标签、推荐结论和证据点都必须尽量清晰，不应只输出一个黑盒分数。

---

## 11. 本地部署方案

本项目本期只在本地运行，建议采用以下方式：

### 前端

本地启动 Next.js 开发服务。

### 后端

本地启动 FastAPI 服务。

### 数据库

## 13. 非功能要求

### 13.1 可演示

第一优先级是把链路打通，确保 Demo 可以完整演示从职位创建到候选人分析再到反馈处理的主流程。

### 13.2 可维护

虽然是 Demo，但代码仍应有基本模块边界，避免所有逻辑堆在单个接口或页面中。

### 13.3 可扩展

本期虽然只使用 SQLite 和本地文件存储，但设计上应支持后续切换到更正式的生产架构。

### 13.4 可调试

本地运行时需要便于查看请求、数据库记录和 AI 输出结果，方便迭代 prompt 和业务逻辑。

---

## 14. 后续扩展方向

如果本期 Demo 验证通过，后续可以继续扩展：

- 从 SQLite 迁移到 PostgreSQL

## 15. 当前架构结论

本项目当前最合适的技术架构是：

- 使用 **Next.js + TypeScript + Tailwind CSS** 构建响应式 Web 前端
    
- 使用 **FastAPI** 承载业务 API 和 AI 工作流
    
- 使用 **SQLite** 作为本地文件数据库
    
- 使用 **LangChain + OpenAI** 实现 AI 分析与生成能力
    
- 使用 **本地文件目录** 存储上传文件
    

这一架构的核心特点不是复杂，而是足够清晰、足够轻、足够适合本地 Demo 快速落地，同时也保留了后续扩展的基本路径。
