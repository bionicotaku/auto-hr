# 后端实现规范文档

## 1. 文档目标

本文档用于定义 `apps/api` 的详细实现规范，确保后端在以下方面保持一致性：

- 项目结构
- 分层边界
- ORM / Schema 组织
- 事务规范
- AI Workflow 实现方式
- 文件处理规则
- API 设计风格
- 错误处理、日志与测试规范

本文档是后端开发与 Code Review 的直接依据。

---

## 2. 后端固定前提

### 2.1 技术栈

- FastAPI
- Python 3.11+
- SQLAlchemy 2.0
- Alembic
- SQLite
- LangChain + OpenAI Responses API

### 2.2 Python 运行环境约定

- 仓库根目录 `./.venv` 是唯一允许使用的 Python 虚拟环境
- `apps/api` 内不再单独创建 `.venv`、`venv` 或其他 Python 环境目录
- 后端开发、测试、迁移、脚本执行都默认通过仓库根目录 `./.venv` 运行

推荐命令写法：

```bash
./.venv/bin/python -m uvicorn app.main:app --reload
./.venv/bin/pytest
./.venv/bin/alembic upgrade head
```

### 2.3 固定业务边界

后端只服务于当前 Demo 主链路：

**Job draft 创建 → B 页面编辑 → 最终定稿 → Candidate 导入 → Candidate 标准化 → rubric 逐项分析 → supervisor 汇总 → 反馈 → 邮件草稿**

明确不做：

- 复杂权限系统
- 多租户
- 分布式任务队列
- Candidate 分析失败中间态持久化
- Job 规范版本系统

### 2.4 固定事务规则

#### Job

- 首次生成可以创建 `draft` Job
- `draft` 不算正式生效内容
- 最终定稿成功后才写入当前生效内容
- 新建流程取消时删除 draft

#### Candidate

- Candidate 全成功才入库
- 任一步失败不写 Candidate 相关记录
- 失败时清理临时文件

## 4. 分层边界规范

## 4.1 API 层

API 层只负责：

- 解析请求
- 调用 Service
- 转换响应

API 层禁止：

- 直接写 ORM
- 直接拼 prompt
- 直接调用 OpenAI
- 直接处理复杂事务

## 4.2 Service 层

Service 层是业务主入口。

职责：

- 协调 repository
- 协调 workflow
- 协调文件处理
- 控制事务边界

Service 层禁止：

- 直接定义 HTTP 语义
- 内联大段 prompt
- 把数据库查询细节散落在业务逻辑中

## 4.3 Repository 层

Repository 只负责数据库读写。

职责：

- 查询
- 插入
- 更新
- 删除

禁止：

- 承担业务判断
- 调用 AI
- 读取文件

## 4.4 Workflow 层

Workflow 层只负责 AI 流程本身。

职责：

- 输入组装
- 模型调用
- schema 校验
- 输出转换

禁止：

- 直接事务落库
- 直接暴露给 router

## 5. 文件结构规范

## 5.1 Router 文件

一个资源一个 router 文件：

- `jobs.py`
- `candidates.py`
- `feedbacks.py`
- `email_drafts.py`

禁止：

- 一个 `api.py` 塞满所有路由

## 5.2 Model 文件

一张主表一个文件：

- `job.py`
- `job_rubric_item.py`
- `candidate.py`
- `candidate_profile.py`
- `candidate_document.py`
- `candidate_rubric_result.py`
- `candidate_tag.py`
- `candidate_feedback.py`
- `candidate_email_draft.py`

## 5.3 Repository 文件

按聚合拆分：

- `job_repository.py`
- `candidate_repository.py`
- `feedback_repository.py`
- `email_draft_repository.py`

## 5.4 Service 文件

按业务用例拆分：

- `job_service.py`
- `candidate_import_service.py`
- `candidate_analysis_service.py`
- `candidate_scoring_service.py`
- `candidate_supervisor_service.py`
- `feedback_service.py`
- `email_draft_service.py`

## 5.5 Workflow 文件

必须按主链路分目录：

```text
workflows/
  job_definition/
  candidate_analysis/
```

禁止：

- 在一个文件里堆所有 AI 流程

## 6. 命名规范

## 6.1 Python 命名

- 文件：`snake_case.py`
- 类：`PascalCase`
- 函数 / 变量：`snake_case`
- 常量：`UPPER_SNAKE_CASE`

## 6.2 类命名

必须带职责后缀：

- `JobService`
- `CandidateRepository`
- `CandidateStandardizeWorkflow`
- `OpenAIResponsesClient`

禁止：

- `Manager`
- `Handler`
- `Processor`

除非职责极其明确。

## 6.3 Schema 命名

- API 输入：`CreateJobFromDescriptionRequest`
- API 输出：`CandidateDetailResponse`
- AI 输出：`CandidateSupervisorSchema`

必须区分：

- API Schema
- ORM Model
- AI Schema

禁止混用。

---

## 7. ORM 与数据模型规范

## 7.1 ORM 原则

- SQLAlchemy Model 只描述持久化结构
- 不把复杂业务方法塞进 Model
- 不把 Pydantic 校验写到 ORM Model 里

## 7.2 JSON 字段原则

适合 JSON TEXT 的内容：

- Job 结构化信息
- 最近对话
- rubric 评分标准
- Candidate 工作经历
- Candidate 教育经历
- Candidate 技能
- Candidate 申请问答
- 证据点列表

不该塞 JSON 的内容：

- Candidate 逐项分析结果集合
- Candidate 标签
- Candidate 反馈
- Email 草稿

## 7.3 Job 生命周期字段

`jobs.lifecycle_status` 固定：

- `draft`
- `active`

禁止：

- 用布尔值代替
- 用模糊字符串扩散状态

---

## 8. API 设计规范

## 8.1 路由设计原则

- 使用资源导向 URL
- URL 表达资源关系，不表达实现细节
- 动作用 `POST/PATCH/DELETE` 体现

## 8.2 响应设计原则

必须：

- 返回稳定字段
- 返回结构化错误
- 不泄露内部异常栈

## 8.3 输入校验原则

所有 API 输入都必须通过 Pydantic 校验。

必须校验：

- 必填
- 枚举
- 数值范围
- 文件数量上限

## 8.4 错误码与错误信息

建议统一响应结构：

```json
{
  "code": "candidate_analysis_failed",
  "message": "Candidate analysis failed.",
  "details": {}
}
```

禁止：

- 返回裸字符串
- 返回前端无法识别的随机错误结构

---

## 9. AI 实现规范

## 9.1 统一 AI 客户端

所有 AI 调用必须统一走：

- LangChain + OpenAI Responses API
- `gpt-5.4`
- `reasoning = medium`
- `.env` 中的 `OPENAI_API_KEY`
- `text.format` + JSON Schema

禁止：

- 某个 workflow 自己手写另一套 OpenAI 调用
- 某个子 agent 私自换模型
- 返回自由文本再手动 parse

## 9.2 AI 输出规范

所有输出必须：

- 可被 schema 校验
- 枚举值可控
- 数值范围可控
- 入库前再次验证

## 9.3 Prompt 文件规范

prompt 只能放在 `app/ai/prompts/`。

禁止：

- prompt 内联在 service
- prompt 内联在 router
- prompt 分散在 util 文件里

## 9.4 Workflow 分阶段规范

### Job Workflow

- draft create
- chat
- agent edit
- regenerate
- finalize

### Candidate Workflow

- import prepare
- standardize
- score items
- summarize
- persist

每一阶段都必须：

- 输入明确
- 输出明确
- schema 明确

---

## 10. Job 业务实现规范

## 10.1 Draft 创建

首次生成成功后：

- 创建 `draft` Job
- 写入 rubric items
- 返回 `job_id`

## 10.2 编辑与对话

普通对话与 Agent 模式：

- 输入当前 JD + 当前 rubric

重新生成：

- 不输入当前 JD
- 不输入当前 rubric
- 只基于原始输入和历史上下文

## 10.3 完成定稿

点击“完成”后：

- 必须再走一次最终定稿 workflow
- 成功后更新 active 内容
- 失败则不更新

## 10.4 取消

新建流程取消：

- 删除 draft Job

编辑 active Job 取消：

- 不改原记录

---

## 11. Candidate 业务实现规范

## 11.1 导入规则

Candidate 导入请求必须满足：

- 文本和文件至少有一项
- PDF 最多 4 个

## 11.2 成功才入库

Candidate 分析流程固定：

1. 临时保存文件
2. 提取文本
3. 标准化
4. rubric 逐项分析
5. supervisor 汇总
6. 全部成功后统一入库

任一步失败：

- 不创建 Candidate 相关记录
- 清理临时文件
- 返回错误

## 11.3 详情页数据结构

Candidate 详情返回必须包含：

- 原始输入
- 标准化信息
- rubric 逐项分析结果
- supervisor 汇总结果
- 状态 / 人工标签 / 备注 / 邮件草稿

---

## 12. 文件处理规范

## 12.1 临时目录

Candidate 导入时一律先进入：

```text
data/tmp/candidate-imports/{request_id}/
```

## 12.2 正式目录

Candidate 成功入库后再移动到：

```text
data/uploads/candidates/{candidate_id}/
```

## 12.3 清理规则

失败时必须：

- 清理临时目录
- 不留下孤儿文件

Job draft 删除时如有相关临时文件，也必须清理。

---

## 13. 事务规范

## 13.1 Job 事务

### 创建 draft

- `jobs`
- `job_rubric_items`

必须同一事务。

### finalize

- 更新 `jobs`
- 删除旧 `job_rubric_items`
- 写入新 `job_rubric_items`

必须同一事务。

## 13.2 Candidate 事务

成功时统一事务写入：

- `candidates`
- `candidate_profiles`
- `candidate_documents`
- `candidate_rubric_results`
- `candidate_tags`

反馈、状态、邮件草稿是后续独立事务。

---

## 14. 错误处理与日志规范

## 14.1 异常设计

业务异常必须有明确语义，例如：

- `DraftJobNotFoundError`
- `JobFinalizeFailedError`
- `CandidateAnalysisFailedError`
- `AiSchemaValidationError`

## 14.2 日志原则

必须记录：

- 请求入口
- workflow 开始 / 结束
- schema 校验失败
- 文件清理
- 数据库事务失败

禁止：

- 打印 API key
- 打印完整隐私原文到常规日志

## 14.3 错误可观测性

对 AI 相关失败，日志至少包含：

- workflow 名称
- schema 名称
- 模型名
- 调用阶段

---

## 15. 测试规范

## 15.1 测试目录

```text
tests/
  api/
  services/
  workflows/
  repositories/
```

## 15.2 必测场景

### Job

- 创建 draft 成功
- 新建取消删除 draft
- finalize 成功更新 active
- finalize 失败不覆盖原记录
- regenerate 不输入当前 JD / rubric

### Candidate

- Candidate 全成功才入库
- 标准化失败不入库
- 某个子 agent 失败不入库
- 成功后文件从临时目录转正式目录
- 详情接口返回完整五块数据

## 15.3 AI 测试原则

优先：

- mock AI client
- 验证 schema 输入输出
- 验证 workflow 编排

不要把在线真实模型调用作为主测试路径。

---

## 16. Code Review 检查清单

提交前必须检查：

- 分层是否清晰
- router 是否足够薄
- service 是否承担业务编排
- repository 是否只做数据读写
- workflow 是否只做 AI 流程
- 事务边界是否正确
- Candidate 是否仍然全成功才入库
- Job draft 生命周期是否正确
- AI 是否统一走 Responses API + JSON Schema
- 是否有孤儿文件或中间状态持久化风险

---

## 17. 明确禁止项

后端实现中明确禁止：

- router 直接操作 ORM
- service 内联长 prompt
- workflow 直接写数据库
- Candidate 失败时写部分记录
- 手写自由文本解析代替 JSON Schema
- 单独某个 AI 调用绕开统一 client
- 用一个超级 service 承载全部业务
- 在 util 层堆业务逻辑

---

## 18. 最终原则

后端实现必须始终围绕四件事保持一致：

- 分层明确
- 事务清晰
- AI 调用统一
- 数据状态可解释

只要某段实现让这四点变模糊，就应当立即拆分或重构。
