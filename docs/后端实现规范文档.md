# 后端实现规范文档

## 1. 文档目标

本文档定义 `apps/api` 的当前实现规范，作为后端开发、审查与回归的直接依据。

---

## 2. 当前固定前提

### 2.1 技术栈

- FastAPI
- Python 3.11+
- SQLAlchemy 2.0
- Alembic
- SQLite
- OpenAI Responses API

### 2.2 运行环境

- 仓库根目录 `./.venv` 是唯一 Python 运行环境
- 迁移、测试、启动、脚本统一使用根目录 `./.venv`

### 2.3 当前业务边界

后端只服务当前 Demo 主链路：

**Job draft → Job 编辑 → Job 保存 → Candidate 分析 → Candidate 展示 → 反馈与邮件草稿**

不做：

- 分布式任务队列
- 复杂权限系统
- Candidate 失败中间态持久化
- Job 评估规范版本管理

---

## 3. 分层规范

### 3.1 Router

只负责：

- 接收请求
- 调用 service
- 返回 response

禁止：

- 直接写 ORM
- 直接调用 OpenAI
- 直接拼复杂事务

### 3.2 Service

负责：

- 业务事务
- 协调 repository
- 协调 workflow
- 协调文件落盘与移动
- 协调 AnalysisRun 状态与事件

### 3.3 Workflow

负责：

- prompt 输入组装
- AI schema 输出校验
- 单阶段业务语义

### 3.4 Repository

只负责数据库读写，不承载业务编排语义。

---

## 4. 长耗时链路规范

### 4.1 AnalysisRun

所有长耗时 AI 主链路统一走：

- `analysis_runs`
- `analysis_run_events`
- `GET /api/analysis-runs/{run_id}`
- `GET /api/analysis-runs/{run_id}/events`

### 4.2 当前 run 类型

- `job_finalize`
- `candidate_import`

### 4.3 前端交互要求

前端不等待单个同步长请求完成。  
后端必须：

- 先创建 run
- 立即返回 `run_id`
- 在后台执行
- 持续写入事件流

---

## 5. Job 规范

### 5.1 当前接口

- `POST /api/jobs/from-description`
- `POST /api/jobs/from-form`
- `GET /api/jobs`
- `GET /api/jobs/{job_id}`
- `GET /api/jobs/{job_id}/edit`
- `POST /api/jobs/{job_id}/chat`
- `POST /api/jobs/{job_id}/agent-edit`
- `POST /api/jobs/{job_id}/finalize-runs`
- `DELETE /api/jobs/{job_id}/draft`

### 5.2 当前语义

- Job draft 首次生成可落库为 `draft`
- `chat` 只返回建议文本
- `agent-edit` 覆盖当前编辑态
- `finalize-runs` 启动 `job_finalize`
- Job 保存成功后写入 `active` 当前真值

### 5.3 Job finalize

`job_finalize` 不再是同步 finalize API。

当前固定规则：

- 标题与摘要重新生成
- rubric 正式富化字段逐项生成
- `weight_normalized` 由代码计算
- progress 按真实 AI 步数推进

---

## 6. Candidate 规范

### 6.1 当前接口

- `GET /api/jobs/{job_id}/candidate-import-context`
- `GET /api/jobs/{job_id}/candidates`
- `POST /api/jobs/{job_id}/candidate-import-runs`
- `GET /api/candidates/{candidate_id}`
- `GET /api/candidates/{candidate_id}/documents/{document_id}/file`
- `PATCH /api/candidates/{candidate_id}/status`
- `POST /api/candidates/{candidate_id}/tags`
- `POST /api/candidates/{candidate_id}/feedbacks`
- `POST /api/candidates/{candidate_id}/email-drafts`

### 6.2 Candidate 入库门槛

Candidate 允许入库的条件只有：

- 分析链路成功
- `is_candidate_like = true`
- 有效姓名存在

以下内容不阻断入库：

- `hard_requirement_overall`
- `recommendation`

### 6.3 默认标签

系统补充默认 AI 标签：

- `has_fail` → `硬性要求未通过`
- `has_borderline` → `需要复核`

### 6.4 Candidate 文件

Candidate 文件链路固定为：

- 保留原始 PDF
- 不提取、不保存 PDF 文本内容
- `candidate_documents.storage_path` 只在后端内部使用
- 对外通过文件接口返回可直接打开的 PDF
- 转正文件名格式为 `[候选人姓名]-[标号].pdf`

---

## 7. 文件处理规范

### 7.1 临时目录

Candidate 上传文件先进入临时目录。

### 7.2 正式目录

分析成功后，文件移动到正式目录并重命名。

### 7.3 对外访问

前端不能直接读取本地路径。  
统一通过：

- `GET /api/candidates/{candidate_id}/documents/{document_id}/file`

---

## 8. 日志与异常

### 8.1 日志

关键 service/workflow 必须记录：

- stage start
- stage success
- stage failure

### 8.2 错误返回

- 请求校验失败要返回具体原因
- 业务错误要返回明确可读信息
- 未知错误保持泛化，不暴露内部实现细节

---

## 9. 当前禁止项

- 不保留旧同步 finalize/import route
- 不保留旧的 5 分总分字段
- 不保留 PDF 文本内容持久化
- 不把 Candidate 原始文件路径直接暴露给前端
- 不让硬门槛冲突阻断 Candidate 入库
