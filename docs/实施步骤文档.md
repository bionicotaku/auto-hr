## 3. 实施总原则

### 3.1 先锁接口与边界，再做页面和流程

必须先锁以下内容：

- monorepo 目录
- 目录 README 分层规则
- Python 虚拟环境
- 环境变量
- 数据模型
- API 契约
- AI schema

否则前后端并行会持续返工。

前端实现阶段还必须同步锁定基础视觉方向，避免各页面各做各的，最后出现功能对齐但视觉割裂。

### 3.2 关键路径优先

关键路径固定为：

**数据库基础设施 → Job draft 主链路 → Job finalize → Candidate 导入主链路 → Candidate 标准化 → rubric 逐项分析 → supervisor 汇总 → Candidate 详情页**

### 3.3 可以并行，但不能并行决定真值

可以并行实施：

- 页面 UI 组件
- DTO 类型文件
- Repository 层
- Alembic 迁移
- 前端 query hooks
- 测试文件

不能并行各自“拍脑袋”决定：

- 数据库字段
- API 返回结构
- AI schema
- Job finalize 语义
- Candidate 全成功才入库事务

### 3.4 子 agent 只适合做边界清晰的子任务

本项目允许使用子 agent 并行实施，但只能在以下条件下使用：

- 任务边界明确
- 文件写入范围清晰
- 不影响当前关键路径判断
- 不需要重新定义数据契约

所有涉及 Python 的实现、测试、迁移和脚本运行，统一使用仓库根目录 `./.venv`。子 agent 不得在子目录再创建独立 Python 虚拟环境。

## 4.2 不建议委派的模块

以下模块应由主实施工程师主导：

- `apps/api/app/core/config.py`
- `apps/api/app/core/db.py`

- 它们直接控制事务、AI 调用和主链路成败
- 任何偏差都会让前后端全部失配

| Repository | 是 | 纯数据库层 | `apps/api/app/repositories/*` |
| Job Workflow | 否 | 关键业务语义 | `apps/api/app/workflows/job_definition/*` |
| Candidate Workflow | 否 | 关键 AI 主链路 | `apps/api/app/workflows/candidate_analysis/*` |

| Candidate 页面组件 | 是 | 边界明确 | `apps/web/components/candidate/*` |

| 测试文件 | 是 | 可跟随主实现并行补齐 | `tests/*` |

## 4.4 子 agent 调用规范

子 agent 不是“把任务甩出去”，而是“在已锁定边界内并行执行”。主调用者必须给足上下文、限制写入范围、要求自验收；子 agent 必须按规范实现并返回明确结论；主调用者必须复验收后，才能把结果视为最终完成。

### 4.4.1 委派前提

只有在以下条件同时满足时，才允许调用子 agent：

- 目标明确，且不是探索性问题
- 需要修改的文件范围已经锁定
- 数据契约、API 契约、AI schema、事务语义已经由主调用者确定
- 当前关键路径不依赖子 agent 先做设计判断

如果任务仍处于“需要先决定怎么设计”的状态，不能先叫子 agent 去拍板。

### 4.4.2 主调用者必须提供的上下文

主调用者给子 agent 的 prompt 至少必须包含以下信息：

- 任务目标
- 业务背景
- 当前阶段
- 参考文档
- 相关文件路径
- 允许写入的文件范围
- 禁止修改的文件范围
- 必须遵循的实现规范
- 需要补齐的测试
- 验收标准

如果任务依赖最新设计文档，必须在 prompt 中明确指出“以哪个文档为准，发生冲突时如何裁决”。不能默认子 agent 自己猜优先级。

### 4.4.3 推荐 prompt 结构

建议主调用者按以下结构组织 prompt：

1. 任务目标
2. 背景与当前固定设计
3. 允许写入的文件
4. 不允许修改的文件
5. 具体实现要求
6. 注释与代码风格要求
7. 需要补的测试
8. 完成后必须返回的结论

推荐明确写入以下要求：

- 严格遵循 [工程实现文档.md](/Users/evan/Downloads/auto-hr/docs/工程实现文档.md)
- 严格遵循 [前端实现规范文档.md](/Users/evan/Downloads/auto-hr/docs/前端实现规范文档.md) 或 [后端实现规范文档.md](/Users/evan/Downloads/auto-hr/docs/后端实现规范文档.md)
- 不要自行扩展 scope
- 不要改动未授权文件
- 不要重定义字段、接口、schema 或事务语义
- 不要跳过测试

### 4.4.4 prompt 必须足够具体

主调用者给子 agent 的 prompt 不能只写：

- “实现这个模块”
- “按文档完成”
- “补一下测试”

这种 prompt 信息不够，会导致子 agent 自己补设计。

主调用者应尽量把任务写成可以直接执行的形式，例如：

- 目标页面或模块是什么
- 依赖哪些已有接口或数据结构
- 要新增哪些文件
- 要修改哪些现有文件
- 页面交互或后端行为必须满足什么条件
- 要覆盖哪些边界场景
- 测试至少断言哪些结果

### 4.4.5 子 agent 的实现约束

子 agent 实施时必须遵守以下约束：

- 严格限制在授权写入范围内
- 不得擅自修改关键语义文件
- 不得自行改变目录结构
- 不得自行引入新的基础库，除非主调用者明确允许
- 不得为省事绕过既定分层
- 不得用模糊命名和临时字段名

前端实现必须遵循：

- 组件职责单一
- 状态来源清晰
- API 调用统一封装
- 不在展示组件里混入接口真值判断
- 保持 TypeScript 类型和 DTO 一致

后端实现必须遵循：

- router / service / repository / workflow 分层清晰
- 事务边界明确
- schema 与 ORM 职责分离
- AI 调用统一走既定 client
- 错误处理与日志结构一致

### 4.4.6 子 agent 的注释要求

子 agent 不能为了“写得快”完全不写注释。以下情况必须补简洁注释：

- 事务边界不直观
- AI workflow 分支较复杂
- 某个转换逻辑依赖设计文档中的特殊规则
- 某个前端状态流有特殊约束
- 某个测试用例在防回归，而不是在验证表面输出

注释要求：

- 只解释非显然逻辑
- 注释要短，说明“为什么”而不是重复“做了什么”
- 不要写无意义注释

### 4.4.7 子 agent 的自验收要求

子 agent 在返回前必须自行完成一轮验收，至少包括：

- 运行与自己改动相关的测试
- 如果没有完整测试，也要运行最小可执行验证
- 检查导入、类型、路径、命名是否一致
- 检查是否越权修改了未授权文件
- 检查是否违反实现规范文档

子 agent 不能只回“已完成”，必须明确说明：

- 修改了哪些文件
- 运行了哪些测试或验证命令
- 测试结果如何
- 是否存在未解决风险
- 结论是“已通过自验收”还是“未通过自验收”

### 4.4.8 子 agent 的返回格式要求

子 agent 完成后，建议按以下结构返回：

1. 修改内容摘要
2. 修改文件列表
3. 测试与验证结果
4. 风险与未完成项
5. 自验收结论

如果某项无法完成，必须明确指出阻塞原因，不能用模糊表述带过。

### 4.4.9 主调用者的复验收责任

子 agent 返回“已通过自验收”，不等于任务已经最终完成。主调用者必须再做一轮复验收，至少包括：

- 复查子 agent 实际改动的文件
- 检查是否符合当前阶段目标
- 检查是否越权修改文件
- 检查是否违反项目实现规范
- 运行关键测试或最小联调验证
- 必要时补修边角问题

只有主调用者复验收通过后，才能向上游汇报“该步骤已完成”。

### 4.4.10 主调用者不得省略最终验收

以下做法是不允许的：

- 子 agent 一返回就直接合并结论
- 没看改动文件就认为任务完成
- 没跑测试就把子 agent 结果当最终结果
- 发现偏离设计后不修正，直接带着问题进入下一阶段

主调用者必须对最终结果负责。子 agent 负责并行执行，不负责替代主调用者做最终判断。

## 22. 最终建议

实际实施时，不要把“写文档中的阶段”误解成必须完全串行。

正确做法是：

1. 主实施工程师先锁定关键契约
2. 把可并行、写入范围清晰的模块分给子 agent
3. 主实施工程师始终控制：
   - 数据模型真值
   - API 真值
   - AI schema 真值
   - 事务真值
4. 每完成一个阶段，就做该阶段自己的测试和验收，不要等到最后一起赌联调

这个项目最怕的不是“实现慢”，而是：

- 前后端各自实现了一套不同契约
- AI workflow 和数据模型脱节
- Candidate 失败入库规则被悄悄破坏
- B 页面编辑真值被前端状态覆盖错位

因此本实施文档的核心要求只有一句话：

**任何并行实施都不能破坏契约真值，任何阶段都必须有对应测试和验收。**
