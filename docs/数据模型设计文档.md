# 数据模型设计文档

## 1. 文档目标

本文档定义当前 Demo 的数据库真值，包括：

- 核心实体
- 表关系
- 持久化边界
- 文件与分析运行时模型

本文档以当前单条初始迁移 `20260410_0007_initial_schema` 为准。

---

## 2. 当前固定实体

本期数据模型只覆盖以下对象：

- Job
- JobRubricItem
- Candidate
- CandidateProfile
- CandidateDocument
- CandidateRubricResult
- CandidateTag
- CandidateFeedback
- CandidateEmailDraft
- AnalysisRun
- AnalysisRunEvent

---

## 3. 关键事务语义

### 3.1 Job

- 新建岗位首次生成时允许创建 `draft`
- `draft` 不进入岗位列表
- 点击“保存”后必须先创建 `job_finalize` analysis run
- 只有 run 成功后，Job 当前生效内容才被写入

### 3.2 Candidate

- Candidate 通过 `candidate_import` analysis run 生成
- 只有分析链路成功、资料有效、且存在有效姓名时才入库
- `hard_requirement_overall` 与 `recommendation` 不阻断入库
- Candidate 失败不保留中间数据库记录

---

## 4. 关系总览

```text
Job
  ├── 1:N JobRubricItem
  └── 1:N Candidate

Candidate
  ├── 1:1 CandidateProfile
  ├── 1:N CandidateDocument
  ├── 1:N CandidateRubricResult
  ├── 1:N CandidateTag
  ├── 1:N CandidateFeedback
  └── 1:N CandidateEmailDraft

AnalysisRun
  └── 1:N AnalysisRunEvent
```

当前 Demo 中，Candidate 直接从属于 Job，不做全局人才库复用。

---

## 5. 关键表语义

### 5.1 `jobs`

保存当前 Job 真值：

- `lifecycle_status`
- `creation_mode`
- `title`
- `summary`
- `description_text`
- `structured_info_json`
- 原始输入与编辑态上下文

### 5.2 `job_rubric_items`

保存当前唯一生效的 rubric 项：

- `sort_order`
- `name`
- `description`
- `criterion_type`
- `weight_input`
- `weight_normalized`
- `scoring_standard_items_json`
- `agent_prompt_text`
- `evidence_guidance_text`

`scoring_standard_items_json` 是当前真值，不再使用旧的字典式字段。

### 5.3 `candidates`

保存 Candidate 当前聚合结果：

- 标准化后的基本身份信息
- 原始文本输入
- `hard_requirement_overall`
- `overall_score_percent`
- `ai_summary`
- `evidence_points_json`
- `recommendation`
- `current_status`

当前明确不存在：

- 旧的 5 分总分字段

### 5.4 `candidate_profiles`

保存 Candidate 标准化档案中的大块 JSON：

- 工作经历
- 教育经历
- 技能
- 就业偏好
- 申请问答
- 补充信息

### 5.5 `candidate_documents`

保存原始 PDF 文件元数据：

- `filename`
- `storage_path`
- `mime_type`
- `page_count`
- `upload_order`

当前真值：

- 保留原始 PDF
- 不保存 PDF 文本内容
- `storage_path` 是后端内部路径，不直接暴露给前端

### 5.6 `candidate_rubric_results`

保存 Candidate 逐项分析结果：

- weighted 项使用 `score_0_to_5`
- hard requirement 项使用 `hard_requirement_decision`
- `reason_text`
- `evidence_points_json`
- `uncertainty_note`

### 5.7 `candidate_tags`

统一保存：

- supervisor 返回的 AI 标签
- 系统补充的默认硬门槛标签
- 人工标签

### 5.8 `candidate_feedbacks`

保存人工备注，采用 append-only 记录方式。

### 5.9 `candidate_email_drafts`

保存邮件草稿，不做真实发送记录。

### 5.10 `analysis_runs`

保存长耗时 AI 运行状态快照：

- `run_type`
- `status`
- `current_stage`
- `current_ai_step`
- `total_ai_steps`
- `result_resource_type`
- `result_resource_id`
- `error_message`
- `payload_json`

当前 run 类型固定为：

- `job_finalize`
- `candidate_import`

### 5.11 `analysis_run_events`

保存 run 事件流，供前端 SSE 消费。

---

## 6. 当前固定数据语义

### 6.1 Job 只有一套当前生效规范

本期不做 Job 评估规范版本管理。  
Job 被重新编辑并保存成功后，当前内容直接覆盖。

### 6.2 Candidate 原始文件只保留原件

当前只保留：

- 原始文本 `raw_text_input`
- 原始 PDF 文件元数据

不保存：

- PDF 提取文本

### 6.3 Candidate 入库与硬门槛

`hard_requirement_overall` 只用于：

- 详情页展示
- 默认标签规则
- 列表筛选

不用于阻断 Candidate 入库。

### 6.4 AnalysisRun 是当前长耗时主链路真值

长耗时 AI 主链路不再依赖同步 HTTP 长请求。  
统一通过 `analysis_runs` 与 `analysis_run_events` 建模和驱动前端进度。
