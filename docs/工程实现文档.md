# 工程实现文档

## 1. 文档目标

本文档用于把当前仓库内已有的需求、设计、技术架构、数据模型、页面路由与 AI 工作流文档，收敛为一份可以直接指导落地开发的工程实现文档。

本文档关注：

- monorepo 目录结构
- 前后端模块划分
- 数据库与文件目录落地方式
- AI 调用统一实现方式
- 每条主业务链路的工程实现步骤
- 页面到 API、API 到 Service、Service 到 Workflow、Workflow 到数据落库的映射

本文档不再讨论备选方案，只描述当前固定实现方案。

---

## 2. 当前固定前提

### 2.1 产品边界

当前 Demo 只覆盖以下闭环：

**Job 创建与编辑 → JD + rubric 评估规范生成/维护 → Candidate 导入 → Candidate 标准化 → rubric 逐项分析 → supervisor 汇总 → Candidate 展示 → 反馈与邮件草稿**

明确不做：

- 自动发布职位
- 自动抓取候选人
- ATS 全量替代
- 真正的异步任务系统
- Candidate 分析失败中间态持久化
- Job 评估规范版本管理

### 2.2 技术栈

- 前端：Next.js App Router + TypeScript + Tailwind CSS + React Query
- 后端：FastAPI + Python 3.11+
- ORM / 迁移：SQLAlchemy 2.0 + Alembic
- 数据库：SQLite
- AI：LangChain + OpenAI Responses API
- 模型：`gpt-5.4`
- 推理强度：`reasoning = high`
- 文件存储：本地目录

### 2.3 前端视觉实现要求

前端除了满足功能闭环，还必须满足以下实现要求：

- 页面尽量保证美观
- 整体观感尽量现代
- 同一 Demo 内视觉风格保持一致

这条要求会直接影响：

- 基础 UI 组件设计
- 页面布局与区块层次
- 颜色、间距、字号、圆角、阴影等视觉 token
- 前端开发阶段验收标准

### 2.3.1 前端展示信息约束

前端页面属于产品交付面，不是技术说明面。

所有用户可见页面、业务组件、空状态和错误提示都必须遵循以下规则：

- 只展示用户完成当前任务所需的信息
- 只展示用户需要理解的状态和操作
- 不展示技术实现细节
- 不展示工程阶段或开发进度说明

明确禁止展示的内容包括：

- 技术栈
- 架构分层
- 状态管理方案
- 数据库方案
- AI 调用方式
- API 实现细节
- “骨架 / 占位 / 当前阶段 / 后续会”一类开发中说明

如需解释内部实现，只能写入 `docs/` 或各工程目录自己的 `README.md`。

### 2.4 AI 固定约束

所有 AI 调用统一遵循：

- 统一通过 LangChain + OpenAI Responses API 发起
- 统一使用 `gpt-5.4`
- 统一使用 `reasoning = high`
- 统一从 `.env` 读取 `OPENAI_API_KEY`
- 统一要求严格 JSON Schema 输出
- 在 Responses API 中统一使用 `text.format`
- 带文件 input 的调用也必须走同一套 Responses API 约束

### 2.4.1 子 agent 固定约束

如果使用子 agent，并行实施必须遵循以下规则：

- 子 agent 统一只允许使用 `gpt-5.4`
- 禁止使用任何 mini 模型
- 禁止为了速度私自切换到其他降级模型
- 是否调用子 agent 由主实施工程师最终判断

文档中的“可委派”“建议委派”只表示该任务具备可委派条件，不构成必须委派。即使某模块可以委派，只要主实施工程师判断本地实现更稳妥、更高效，就应直接本地实现。

### 2.5 事务固定规则

#### Job

- 新建 Job 首次生成后，先创建一条 `draft` 记录
- B 页面点击“完成”后，再发起最终定稿调用
- 最终定稿成功并通过校验后，才更新为当前生效内容
- 新建流程点击“取消”时，删除对应 `draft`
- `draft` 不算正式入库的生效 Job，可定期清理

#### Candidate

- Candidate 导入与分析走全成功才入库
- 任一步失败都不写 Candidate 相关数据库记录
- 失败时清理临时文件，不保留中间状态

### 3.1 目录原则

- `apps/web` 只放前端页面、组件、hooks 和前端请求封装
- `apps/api` 只放 API、数据库、AI Workflow 和文件处理逻辑
- `packages/ui` 只放可复用的前端基础组件，不放业务状态
- `packages/config` 放共享配置模板、根级 lint / format 配置
- `data` 是本地 SQLite 和文件目录，不放业务代码
- `scripts` 放一键启动和初始化脚本
- `docs` 统一放项目需求、设计、架构、规范和实施文档
- 每个关键工程目录保留自己的 `README.md`，用于解释本目录职责、关键文件和局部约束

### 3.3 文档分层规则

- `docs/` 只放项目级设计文档、技术架构文档、实现规范、实施步骤等全局文档
- 各工程目录自己的 `README.md` 留在本目录，作为就地导航和局部说明
- 不把目录 README 挪到 `docs/` 代替本地说明

### 3.4 为什么保持在同一 monorepo

- 业务链路高度耦合，前后端迭代需要一起推进
- 当前是 Demo，分仓会额外增加联调成本
- 文档、接口、AI Workflow、前端页面在同一仓库更容易保持同步
- 本地 SQLite、上传文件、前后端地址都以本地开发为中心，单仓更合适

## 4. 根目录实现

## 4.1 根配置文件

### `/package.json`

职责：

- 管理 workspace
- 挂载统一脚本入口
- 提供 lint / format / dev 的顶层命令

建议脚本：

- `dev:web`
- `dev:api`
- `dev`
- `lint`
- `format`

### `/pnpm-workspace.yaml`

职责：

- 管理 `apps/web` 和 `packages/*`

### `/.env.example`

职责：

- 提供本地开发所需环境变量模板

建议字段：

```env
OPENAI_API_KEY=
OPENAI_MODEL=gpt-5.4
OPENAI_REASONING=high
DATABASE_URL=sqlite:///./data/app.db
DATA_DIR=./data
UPLOAD_DIR=./data/uploads
TEMP_UPLOAD_DIR=./data/tmp
API_HOST=0.0.0.0
API_PORT=8000
NEXT_PUBLIC_API_BASE_URL=http://localhost:8000
```

### `/Makefile`

职责：

- 把跨语言开发命令统一到一个入口

建议命令：

- `make bootstrap`
- `make dev`
- `make dev-web`
- `make dev-api`
- `make lint`
- `make test`

### `./.venv`

职责：

- 作为仓库唯一 Python 虚拟环境
- 统一承载 `apps/api`、根脚本、测试、迁移命令的 Python 运行时

固定约束：

- 不在 `apps/api` 内再创建独立虚拟环境
- 所有 Python 命令默认通过仓库根目录 `./.venv` 执行
- 文档中的 Python 示例命令优先写成 `./.venv/bin/python`、`./.venv/bin/pip`、`./.venv/bin/pytest`、`./.venv/bin/alembic`

推荐示例：

```bash
./scripts/bootstrap.sh
./.venv/bin/python -V
./.venv/bin/pip install -e apps/api
./.venv/bin/pytest
```

## 6.5 Service 层

Service 是业务编排层，负责协调 Repository、Workflow、文件处理和事务。

### `job_service.py`

职责：

- 从原始描述创建 draft Job
- 从基础表单创建 draft Job
- 获取 Job draft 详情
- 处理 B 页面普通对话
- 处理 Agent 编辑
- 处理重新生成
- 处理最终定稿
- 删除 draft Job

### `candidate_import_service.py`

职责：

- 校验 Candidate 导入输入
- 创建临时处理上下文
- 调用 CandidateAnalysisService
- 成功后统一落库
- 失败时回滚并清理临时文件

### `candidate_analysis_service.py`

职责：

- 编排标准化
- 编排 rubric 逐项评分
- 编排 supervisor 汇总
- 返回可落库的完整分析产物

### `candidate_scoring_service.py`

职责：

### Candidate 分析 Workflow

目录：


#### `import_prepare.py`

- 提取 PDF 文本
- 组装分析输入

#### `standardize.py`

职责：

- 使用文件 input + 文本 input 做 Candidate 标准化

## 7.3 Job 生命周期

### 新建

1. 用户提交“已有描述导入”或“基础信息生成”
2. 后端首次 AI 生成成功
3. 创建 `jobs.lifecycle_status = draft`
4. 写入对应 `job_rubric_items`
5. 返回 `jobId`
6. 前端跳转编辑页

### 编辑完成

1. 用户点击“完成”
2. 后端执行最终定稿调用
3. 成功后事务更新：
   - `jobs`
   - `job_rubric_items`
4. `lifecycle_status` 设为 `active`

### 新建取消

1. 用户在新建流程点击“取消”
2. 删除对应 `draft` Job 和 `job_rubric_items`

## 9. 关键业务链路实现

## 9.1 Job 创建链路

### 从描述创建

1. 前端输入原始职位描述
2. 调用 `POST /api/jobs/from-description`
3. 后端调用 Job draft create workflow
4. 生成 JD 草稿 + rubric 草稿
5. 创建 `draft` Job
6. 返回 `jobId`
7. 跳转 `/jobs/[jobId]/edit`

### 从表单创建

1. 前端输入基础字段
2. 调用 `POST /api/jobs/from-form`
3. 后端生成 JD 草稿 + rubric 草稿
4. 创建 `draft` Job
5. 返回 `jobId`
6. 跳转编辑页

---

## 9.2 B 页面链路

### 普通对话

输入：

- 当前 JD
- 当前 rubric
- 历史摘要
- 最近 5 轮对话
- 当前用户输入

输出：

- 建议文本

不覆盖编辑区。

### Agent 模式

输入与普通对话类似。

输出：

- 新 JD
- 新 rubric

成功后覆盖当前编辑区。

### 重新生成

输入：

- 原始输入
- 历史摘要
- 最近 5 轮对话
- 必要上下文

明确不输入：

- 当前 JD
- 当前 rubric

目的：

- 避免模型刻意参考当前版本
- 避免模型为了“避开当前内容”产生不自然结果

### 完成

1. 前端点击“完成”
2. 调用 `POST /api/jobs/{job_id}/finalize`
3. 后端执行最终定稿 workflow
4. 成功后更新 Job 生效内容
5. 跳转 `/jobs/[jobId]`

### 取消

- 新建流程：删除 draft
- 编辑流程：直接返回，不覆盖 active Job

---

## 9.3 Candidate 导入与分析链路

1. 用户进入 `/jobs/[jobId]/candidates/new`
2. 输入文本和/或选择 PDF
3. 调用导入接口
4. 后端创建临时目录并保存上传文件
5. 提取 PDF 文本
6. 执行 Candidate 标准化
7. 按 rubric 拆分子任务
8. 执行多个子 agent 分析
9. 执行 supervisor 汇总
10. 成功后统一落库
11. 返回 `candidateId`
12. 跳转 `/candidates/[candidateId]`

失败时：

- 返回错误
- 前端保留当前输入
- 后端清理临时文件
- 不写数据库

## 12. 当前推荐结论

当前项目最适合直接在同一个 monorepo 中落地：

- `apps/web` 承载前端页面与交互
- `apps/api` 承载业务 API、数据库和 AI Workflow
- `data` 承载 SQLite、正式上传文件与临时导入文件
- 根目录保留所有设计文档，持续与实现同步

这份工程方案的核心不是做成“大而全”的平台，而是保证当前 Demo 能稳定打通以下关键路径：

**Job draft 创建 → B 页面编辑 → 最终定稿 → Candidate 导入 → Candidate 标准化 → rubric 逐项分析 → supervisor 汇总 → Candidate 展示 → 反馈与邮件草稿**
