# 前端实现规范文档

## 1. 文档目标

本文档定义 `apps/web` 的当前实现规范，确保前端页面、状态、交互和 API 使用方式与现有实现一致。

---

## 2. 当前固定前提

### 2.1 技术栈

- Next.js App Router
- TypeScript
- Tailwind CSS
- React Query

### 2.2 视觉体系

- 真实业务页统一采用 `/ui-lab` 对应的 light-first glass 风格
- 真实业务页必须同时支持 light / dark
- 真实业务页默认跟随系统主题
- 真实业务页右下角统一提供悬浮 sidebar，包含“返回上一页”和“浅色 / 深色模式切换”
- 手动主题切换只在当前会话内生效，刷新后重新回到系统主题
- 不保存用户主题偏好
- 页面背景必须提供足够的 layered backdrop，使透明卡片始终有可感知的折射层次，而不是贴在单一底色上
- `/ui-lab` 保留为视觉参考页，不作为真实业务页的实现来源

### 2.3 主链路

前端围绕以下闭环组织：

**Job 列表 → Job 新建 → Job 编辑 → Job 保存 → Candidate 分析 → Candidate 详情 → 状态/标签/备注/邮件草稿**

### 2.4 展示面约束

前端页面只展示用户完成当前任务需要的信息。  
禁止在用户界面暴露：

- 技术栈
- 架构分层
- API 实现细节
- 数据库字段
- “当前阶段”“占位”“后续会”之类开发说明

---

## 3. 路由真值

当前固定路由：

```text
/jobs
/jobs/new
/jobs/new/from-description
/jobs/new/from-form
/jobs/[jobId]
/jobs/[jobId]/edit
/jobs/[jobId]/candidates/new
/candidates/[candidateId]
```

### 3.1 页面职责

- `/jobs`
  - 只展示已生效岗位
- `/jobs/new/*`
  - 创建 Job draft
- `/jobs/[jobId]/edit`
  - 编辑当前 Job 真值
  - 获取建议、生成新版、保存
- `/jobs/[jobId]/candidates/new`
  - 提交 Candidate 原始文本和 PDF，启动分析
- `/candidates/[candidateId]`
  - 展示标准化信息、原始输入、汇总结论、逐项分析和处理状态

---

## 4. 状态管理真值

### 4.1 服务端状态

来自后端的状态统一由 React Query 管理：

- 列表
- 详情
- analysis run 状态快照
- 各类 mutation 成功后的刷新

### 4.2 页面本地状态

以下内容属于页面本地状态：

- Job 编辑页当前输入
- Candidate 导入页原始文本
- 已选择但尚未提交的文件
- AI 对话输入
- Candidate 列表筛选值
- 备注输入、标签输入、邮件草稿类型选择

### 4.3 Job 详情 Candidate 列表

Candidate 列表筛选、排序、搜索属于页面本地状态，不进入 URL。

固定规则：

- 不使用 `router.push/replace` 同步这些条件
- 不使用 `searchParams` 恢复这些条件
- 刷新页面后恢复默认值
- 搜索输入即时更新，请求值可做轻量 debounce

## 5. Job 编辑页规范

### 5.1 编辑态真值

编辑页本地维护：

- `title`
- `summary`
- `description_text`
- `responsibilities`
- `skills`
- `rubric_items`

初始化完成后，本地状态是当前真值，不允许被远端缓存自动覆盖。

### 5.2 按钮语义

- 获取建议
  - 只追加对话
  - 不覆盖编辑区
- 生成新版
  - 覆盖当前编辑态
- 保存
  - 不直接同步 finalize
  - 必须先创建 `job_finalize` analysis run

### 5.3 保存交互

点击“保存”后，前端必须：

1. 调用 `POST /api/jobs/{jobId}/finalize-runs`
2. 拿到 `run_id`
3. 展示统一进度卡
4. 通过 `SSE` 监听进度
5. run 成功后跳转 `/jobs/[jobId]`

如果是已生效岗位且没有任何修改，则“保存”等同直接返回详情页。

---

## 6. Candidate 导入页规范

### 6.1 表单真值

Candidate 导入页只接收两类输入：

- 原始文本
- PDF 文件

按钮文案固定为“分析”。

### 6.2 文件交互

- 支持点击选择 PDF
- 支持拖拽上传 PDF
- 拖拽到浏览器页面时必须阻止默认打开文件

### 6.3 分析交互

点击“分析”后，前端必须：

1. 调用 `POST /api/jobs/{jobId}/candidate-import-runs`
2. 拿到 `run_id`
3. 展示统一进度卡
4. 通过 `SSE` 监听进度
5. run 成功后跳转 `/candidates/[candidateId]`

---

## 7. Candidate 详情页规范

### 7.1 固定布局顺序

Candidate 详情页固定顺序：

1. 标准化信息
2. 原始输入
3. 汇总结论
4. 逐项分析
5. 处理状态

### 7.2 原始输入区

- 原始文本：展示 `raw_text_input`
- 原始文件：展示 PDF 元数据和“查看原文件”链接

不展示：

- 提取文本
- 服务器本地路径

### 7.3 处理状态区

当前支持：

- 更新状态
- 添加人工标签
- 添加备注
- 生成邮件草稿

## 8. 统一组件规范

### 8.0 共享视觉基础

真实业务页必须优先复用共享视觉层，而不是逐页重复写样式。

当前共享层包括：

- `globals.css` 中的主题 tokens
- `AppShell`
- `FloatingSideBar`
- `components/ui/` 中的基础 primitives

规则：

- 颜色、边框、阴影、glass surface 和文本层级统一从全局 token 出发
- 真实业务页不应继续散落硬编码亮暗主题样式
- 页面布局允许重组，但必须围绕共享 shell 与共享面板模式组织
- 页面回退行为必须显式声明可用的 `backHref`，不能假设浏览器历史一定存在
- active / 已生效类状态优先复用共享 `Badge` 的 mint 语义，不在页面内单独硬编码另一套样式
- 彩色标签统一收敛到共享 `Badge` 的 mint / lime / blue / amber / orange / red / neutral 家族；建议推进、人工复核、分数标识、权重标签和状态标签都优先用这套模板
- 分数型标签可按数值区间映射到上述色阶；例如百分制信号分可以按每 20 分一档从 red 渐变到 mint

### 8.1 进度卡

长耗时 run 统一使用 `AnalysisProgressCard` 展示：

- 当前阶段
- 当前 AI 步数
- 总 AI 步数
- 错误信息

### 8.2 错误态

关键页面统一使用 `ErrorStateCard` 展示失败信息，不自行拼散落错误卡片。

### 8.3 业务页布局方向

当前业务页统一遵循：

- glass shell 页头
- 明确信息块顺序
- review surface 优先
- control panel 与 history panel 分区

典型落点：

- `/jobs` 和 `/jobs/[jobId]` 以 overview + review surfaces 为主
- `/jobs/[jobId]/edit` 以 definition / rubric / AI workspace 为主
- `/jobs/[jobId]/candidates/new` 以 import workspace 为主
- `/candidates/[candidateId]` 以 review panels + action center 为主

---

## 9. 前端对实现的固定要求

- 不保留已删除的同步 finalize/import 调用
- 不保留 Candidate 列表 URL 驱动
- 不展示旧的 5 分总分字段
- 不展示 PDF 文本内容
- 不展示服务器文件路径
- 所有进度 UI 都以 `AnalysisRun + SSE` 为准
