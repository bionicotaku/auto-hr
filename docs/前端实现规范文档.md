# 前端实现规范文档

## 1. 文档目标

本文档用于定义 `apps/web` 的详细实现规范，确保前端在以下方面保持一致性：

- 目录结构
- 路由组织
- 页面与组件边界
- 状态管理方式
- 请求层封装
- UI 设计一致性
- 命名、类型、代码风格
- 错误处理与测试方式

本文档是前端开发与 Code Review 的直接依据。

---

## 2. 前端固定前提

### 2.1 技术栈

- Next.js App Router
- TypeScript
- Tailwind CSS
- React Query

### 2.2 前端服务的业务链路

前端必须围绕以下主链路组织：

**Job 列表 → Job 新建入口 → Job 草稿创建 → B 页面编辑 → Job 保存定稿 → Job 详情 → Candidate 导入 → Candidate 详情 → 反馈与邮件草稿**

### 2.3 核心对象

前端所有页面结构围绕两个核心对象展开：

- Job
- Candidate

禁止按零散功能拆分为孤立页面，例如单独做“邮件页”“标签页”“评分页”。

### 2.4 视觉质量硬约束

前端实现不仅要可用，还必须尽量保证：

- 页面整体美观
- 视觉风格现代
- 同一产品内视觉语言一致

这不是可选优化，而是前端实现与验收的一部分。

### 2.5 用户展示信息硬约束

前端是产品交付面，不是技术说明面。

用户可见页面只能展示以下内容：

- 当前任务需要的信息
- 当前任务相关的状态
- 当前任务可执行的操作
- 对完成当前任务真正有帮助的提示

禁止在用户可见界面中展示：

- 技术栈名称
- 工程阶段说明
- 架构分层说明
- 状态管理方案
- 数据库方案
- AI 工作流实现细节
- 接口实现细节
- 面向开发者的占位说明

以下文案在用户界面中默认禁止出现：

- “Phase”
- “Skeleton”
- “骨架”
- “占位”
- “后续会”
- “当前阶段”
- 其他明显从开发者视角出发的实现说明

如果需要描述系统内部机制，只能写入 `docs/` 或工程目录本地 `README.md`，不能出现在最终用户界面中。

## 3. 目录结构规范

前端目录固定如下：

```text
apps/web/
  README.md
  app/
    layout.tsx
    page.tsx
    jobs/
      page.tsx
      new/
        page.tsx
        from-description/
          page.tsx
        from-form/
          page.tsx
      [jobId]/
        page.tsx
        edit/
          page.tsx
        candidates/
          new/
            page.tsx
    candidates/
      [candidateId]/
        page.tsx
  components/
    job/
    candidate/
    feedback/
    email/
    layout/
    ui/
  hooks/
  lib/
    api/
    query/
    forms/
    utils/
    constants/
    types/
  styles/
  tests/
```

目录 README 规则：

- `apps/web` 及其关键子目录应保留本地 `README.md`
- 本地 README 只说明本目录职责、关键组件和局部约束
- 项目级设计、架构和实施文档不放在前端目录，统一放 `docs/`

### 3.1 `app/`

职责：

- 只放路由页面和 route-level layout
- 页面只负责页面级编排，不直接写复杂业务细节

### 3.2 `components/`

职责：

- 放可复用组件
- 组件按业务域拆分，不按纯视觉拆散

### 3.3 `hooks/`

职责：

- 放页面复用逻辑
- 放复杂本地交互逻辑
- 不放纯请求函数

### 3.4 `lib/api/`

职责：

- 放请求函数
- 放接口 DTO 类型
- 放 API client 封装

### 3.5 `lib/query/`

职责：

- 放 React Query 的 query keys
- 放 query / mutation hooks

### 3.6 `lib/forms/`

职责：

- 放表单默认值
- 放表单 schema
- 放前端表单转换逻辑

### 3.7 `lib/utils/`

职责：

- 放纯函数
- 不允许耦合 React

### 3.8 `components/ui/`

职责：

- 放 Button、Input、Dialog、Badge、Tabs 这类基础组件
- 不放 Job/Candidate 业务语义

## 4. 路由与页面边界规范

路由结构固定为：

```text
/jobs
/jobs/new
/jobs/new/from-description
/jobs/new/from-form
/jobs/[jobId]
/jobs/[jobId]/edit
/jobs/[jobId]/candidates/new
/candidates/[candidateId]
```

### 4.1 页面职责必须单一

每个页面文件只负责：

- 读取参数
- 调用 query hooks
- 编排业务组件
- 管理页面本地状态

禁止在页面里直接：

- 写长段请求逻辑
- 写大段数据转换逻辑
- 写复杂条件渲染分支而不抽组件
- 直接拼接后端 DTO

### 4.2 页面与组件的拆分标准

当页面中出现以下情况之一，必须抽组件：

- 单个 JSX 区块超过 60-80 行
- 有独立标题和明确业务语义的区块
- 需要复用
- 有自己的交互状态

### 4.3 页面命名规范

页面组件使用默认导出即可，但实际组件名应可读，例如：

- `JobsPage`
- `JobEditPage`
- `CandidateDetailPage`

## 5. 页面级实现规范

## 5.1 Job 列表页

职责：

- 展示 Job 卡片列表
- 跳转 Job 详情
- 跳转新建入口

当前固定语义：

- `/jobs` 只展示 `active` Job
- `draft` 不进入岗位列表
- `draft` 视为未完成的临时数据

禁止：

- 在此页承载 Job 编辑状态
- 在此页持有 Candidate 列表缓存的本地镜像

## 5.2 Job 新建入口页

职责：

- 只做路径选择

禁止：

- 缓存任何业务草稿

## 5.3 Job 描述导入页 / 基础表单页

职责：

- 输入原始数据
- 调用创建 draft API
- 成功后跳转编辑页

必须：

- 保留提交失败错误信息
- 提交中禁用重复提交

## 5.4 Job 编辑页

这是前端实现最严格的页面。

职责：

- 左侧 JD 编辑区
- 右侧 rubric 编辑区
- 下方 AI 对话区
- 保存、取消、获取建议、生成新版

必须遵守：

- 当前标题、摘要、JD、responsibilities、skills 与 rubric 一旦初始化完成，就进入本地受控状态
- 本地受控状态不允许被 React Query 返回的数据自动覆盖
- 普通对话与 Agent 模式输入当前最新编辑态
- `获取建议` 只返回建议文本，不覆盖当前编辑内容
- `生成新版` 直接覆盖当前标题、摘要、结构化信息、JD、responsibilities、skills 与 rubric
- 点击“保存”必须先启动 `job_finalize` analysis run，并展示统一进度卡
- 新建流程取消时必须删除 draft
- active Job 取消时必须直接返回详情页
- active Job 保存且无修改时必须等同于取消，不发保存请求

编辑区固定结构：

- 职位描述
- Responsibilities
- Skills
- rubric 项中的评估项 / 权重 / 说明

rubric 区固定约束：

- 不展示类型
- 不展示标准化权重
- 不展示详细标准入口
- `weight = 100` 代表硬要求
- `weight = 1-99` 代表 weighted

禁止：

- 直接把 React Query 返回对象当作受控编辑值
- 用 URL 保存 JD / rubric 当前内容
- 在多个组件中各自维护一份编辑区真值

## 5.5 Job 详情页

职责：

- 展示 Job summary
- 展示 Candidate 列表
- 支持 Candidate 排序与筛选
- 跳转编辑和导入

URL 允许承载：

- 排序项
- 状态筛选
- 标签筛选
- 搜索词

## 5.6 Candidate 导入页

职责：

- 输入原始文本
- 选择 PDF
- 提交导入分析请求

必须：

- 文本和文件至少一项非空
- 最多 4 个 PDF
- 失败时保留当前输入

禁止：

- 提交前写数据库中间状态
- 前端自行模拟 Candidate 临时记录

## 5.7 Candidate 详情页

此页结构必须固定为五个区块：

1. 原始输入区
2. 标准化信息区
3. rubric 逐项分析区
4. supervisor 汇总区
5. 操作区

必须：

- 原始输入区始终展示
- 操作区只负责用户动作，不负责主数据展示
- 页面顶部提供返回所属 Job 的入口

## 6. 组件设计规范

## 6.1 基础原则

- 组件优先小而清晰
- 一次只负责一个业务职责
- 组件 props 必须显式、可读、可推断

### 推荐组件拆分

#### Job 相关

- `JobCard`
- `JobDescriptionEditor`
- `JobRubricEditor`
- `JobAiChatPanel`
- `JobEditActionBar`
- `JobSummaryPanel`

#### Candidate 相关

- `CandidateList`
- `CandidateListFilters`
- `CandidateRawInputPanel`
- `CandidateNormalizedPanel`
- `CandidateRubricResultsPanel`
- `CandidateSupervisorPanel`
- `CandidateActionPanel`

### 6.2 组件命名规范

- 组件文件用 PascalCase
- 组件名和文件名保持一致

例如：

- `CandidateSupervisorPanel.tsx`
- `JobRubricEditor.tsx`

禁止：

- `index.tsx` 到处泛滥
- `panel.tsx`、`box.tsx`、`item.tsx` 这类无语义命名

### 6.3 props 设计规范

优先：

- 显式字段
- 精准类型

避免：

- 传整个大对象再在组件内乱取字段
- 多层透传无语义回调

### 6.4 受控组件规范

编辑器、表单、筛选器一律优先受控实现。

适用对象：

- JD 编辑器
- rubric 编辑器
- Candidate 导入输入框
- 反馈备注输入框

---

## 7. 状态管理规范

## 7.1 状态分类

前端状态只能分三类：

- 服务端状态
- URL 状态
- 页面本地状态

### 服务端状态

统一由 React Query 管理。

包括：

- Job 列表
- Job 详情
- Job draft 详情
- Candidate 列表
- Candidate 详情
- Candidate 标准化信息
- Candidate 逐项分析结果
- Candidate supervisor 汇总结果
- 反馈记录
- 邮件草稿记录

### URL 状态

只能放可恢复状态：

- 排序
- 筛选
- 搜索
- tab

### 本地状态

只能放临时交互状态：

- 编辑器当前值
- AI 对话输入
- 导入页输入
- 未提交文件
- 备注草稿
- loading
- 局部错误

## 7.2 React Query 规范

必须：

- 按资源维度设计 query key
- query key 可枚举、可推导
- mutation 成功后精确失效相关 query

禁止：

- 手工维护第二份服务端镜像状态
- 在组件树多个位置各自 fetch 同一份关键数据且不共享 query key

## 7.3 编辑页状态规则

Job 编辑页必须同时满足：

- 当前 JD 内容只有一份真值
- 当前 rubric 内容只有一份真值
- React Query 返回值只作为初始化数据
- 初始化后由页面本地状态接管

## 7.4 Candidate 导入页状态规则

必须：

- 导入失败后保留输入
- 文件选择状态不进 URL
- 页面离开前不创建本地伪 Candidate

---

## 8. 数据请求与 DTO 规范

## 8.1 请求层目录规范

```text
lib/api/
  client.ts
  jobs.ts
  candidates.ts
  feedback.ts
  email.ts
  types.ts
```

### `client.ts`

职责：

- 封装基础 `fetch`
- 统一 base URL
- 统一错误解析

### 资源文件

每个资源一个文件：

- `jobs.ts`
- `candidates.ts`
- `feedback.ts`
- `email.ts`

禁止：

- 把所有请求都堆到一个 `api.ts`

## 8.2 DTO 规范

前端只依赖 API DTO，不直接拼接后端 ORM 结构概念。

必须：

- 在 `lib/api/types.ts` 定义 DTO
- DTO 命名包含资源语义

例如：

- `JobListItemDto`
- `JobEditDraftDto`
- `CandidateDetailDto`
- `CandidateSupervisorSummaryDto`

---

## 9. TypeScript 规范

## 9.1 类型优先级

优先顺序：

1. 明确接口类型
2. 精确联合类型
3. 受控泛型
4. 最后才是 `unknown`

禁止：

- 随意使用 `any`
- 把后端返回值直接断言为目标类型而不校验

## 9.2 命名规范

- 类型：PascalCase
- 变量：camelCase
- 常量：`UPPER_SNAKE_CASE` 仅用于真正常量
- hooks：`useXxx`
- query keys：`xxxKeys`

## 9.3 布尔命名规范

必须使用可读语义：

- `isSubmitting`
- `isLoadingDraft`
- `hasUnsavedChanges`

避免：

- `loading`
- `flag`
- `statusOk`

---

## 10. UI 与样式规范

## 10.1 样式原则

- 优先 Tailwind 原子类
- 重复模式抽成组件，不抽成杂乱 className 常量
- 页面以业务可读性优先
- 页面需要兼顾美观、现代感和清晰层次，不能只满足“能显示”

## 10.2 布局原则

- 支持响应式
- 同一业务状态只保留一份数据来源
- 窄屏下通过重排适配，不改变路由和数据结构
- 不同页面在布局密度、区块节奏和信息层级上应保持统一产品感

## 10.3 视觉一致性

必须统一：

- 间距尺度
- 标题层级
- 表单控件尺寸
- loading / empty / error 样式
- 卡片、面板、边框、阴影和圆角语言
- 主色、辅色、强调色和交互态颜色规则
- 按钮层级与交互反馈

必须避免：

- 单页风格明显跳脱，像不同产品拼在一起
- 只堆砌默认控件，缺少视觉层次
- 同类区块在不同页面使用完全不同的间距、边框或标题样式
- 颜色、字号、阴影、圆角随意变化

建议：

- 在 `components/ui` 中沉淀统一 Card / Section / Toolbar 结构
- 在早期先确定一套基础视觉 token，再逐页复用

---

## 11. 表单与交互规范

## 11.1 表单校验

所有页面表单都必须有：

- 必填校验
- 前端可读错误提示
- 提交中禁用逻辑

## 11.2 destructive 操作

例如：

- 删除 draft
- 覆盖编辑区

必须：

- 有明确按钮语义
- 不做模糊自动行为

## 11.3 loading 状态

必须在关键异步链路展示 loading：

- Job draft 创建
- Job 生成新版
- Job 最终定稿
- Candidate 导入分析
- 邮件草稿生成

---

## 12. 错误处理规范

## 12.1 错误分层

- 页面级错误：整页加载失败
- 区块级错误：某个子区块请求失败
- 交互级错误：提交失败、保存失败、分析失败

## 12.2 错误展示原则

必须：

- 给出用户可理解的信息
- 不直接暴露后端 traceback
- 保留用户输入

尤其：

- Job 最终定稿失败时保留当前编辑区内容
- Candidate 导入失败时保留当前文本与文件选择

---

## 13. 测试规范

## 13.1 测试层级

至少覆盖：

- 页面渲染测试
- 关键交互测试
- 请求层单元测试

## 13.2 重点测试场景

- 创建 Job draft 成功后跳转编辑页
- Job 编辑页生成新版会覆盖当前编辑态
- Job 保存前先触发最终定稿
- 新建流程取消删除 draft
- Candidate 导入失败时保留输入
- Candidate 详情页五个区块完整展示

---

## 14. Code Review 检查清单

提交前必须检查：

- 页面职责是否单一
- 组件是否按业务语义拆分
- 服务端状态是否都走 React Query
- URL 是否只承载可恢复状态
- 编辑真值是否只有一份
- API 请求是否都走 `lib/api`
- 是否出现 `any`
- 是否保留了失败时的用户输入
- Candidate 详情页结构是否仍与最新设计一致

---

## 15. 明确禁止项

前端实现中明确禁止：

- 直接在页面里写大段 fetch 逻辑
- 组件里偷偷再发一份主数据请求
- 用 URL 保存编辑器内容
- 用多个 state 保存同一份 JD / rubric 真值
- 用本地数组模拟服务端列表并长期同步
- 使用 `any` 逃避建模
- 在 Candidate 导入前创建本地伪记录
- 把业务组件塞进 `components/ui`

---

## 16. 最终原则

前端实现必须始终围绕三件事保持一致：

- 页面结构与业务对象一致
- 状态真值来源清晰
- 编辑、保存、失败与跳转的行为稳定可预期

只要实现让这三点变模糊，就需要回退并重构。
